**backend/app/services/processing.py** — The background processing orchestrator:

```python
async def process_meeting(
    meeting_id: str,
    transcript_text: str,
    provider: LLMProvider,
    user_id: str,
):
```

Steps:
1. Update status to "detecting_type". Render `detect_meeting_type.j2` with transcript (truncated to first 2000 chars). Call `provider.generate_structured(messages, MeetingTypeResult)`. Save meeting_type and confidence to DB.

2. Update status to "summarizing". Render `summarize_meeting.j2` with full transcript. Call `provider.generate_structured(messages, MeetingSummary)`. Save summary as JSONB to DB.

3. Update status to "extracting_actions". Render `extract_action_items.j2` with transcript and summary. Call `provider.generate_structured(messages, ActionItemsResult)`. For each extracted action item: insert into action_items table with meeting_id, user_id, description, owner_name, due_date (default to 7 days from now if null — per CONTEXT.md smart defaults), status="not_started".

4. Update status to "completed", set processed_at.

Error handling:
- Wrap entire function in try/except. On ANY exception: set status="failed", set error_message=str(e), log the full exception with traceback at ERROR level.
- Log each step start/completion at INFO level.
- The transcript is NEVER deleted on failure — it's already saved in storage from Plan 02.
